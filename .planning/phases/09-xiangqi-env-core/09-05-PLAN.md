<objective>
Fix the FEN parser bug causing halfmove_clock to be read incorrectly from WXF 5-field FEN format, then add a regression test to prevent recurrence. This closes UAT Test 8 gap.

**Root Cause:** `from_fen()` in constants.py reads `parts[4]` unconditionally. WXF FEN omits the en passant field (5 fields: pieces/side/castling/halfmove/fullmove), so parts[3]=halfmove and parts[4]=fullmove. The parser reads fullmove (1) as halfmove, causing halfmove=0 or 1 instead of the actual value (120). This breaks the 50-move rule check (`halfmove_clock >= 100`).

**Bug Evidence:**
```
WXF 5-field: '... w - 120 1'  -> parts[4]='1' (fullmove) -> halfmove=1 WRONG
Standard 6:  '... w - - 120 1' -> parts[4]='120'          -> halfmove=120 CORRECT
```

## Bug Analysis (from empirical testing)

| FEN Format | Example | parts[3] | parts[4] | Current (parts[4]) | Expected |
|------------|---------|----------|----------|---------------------|----------|
| WXF 5-field | `w - 120 1` | `120` | `1` | 1 (WRONG) | 120 |
| Standard 6-field | `w - - 120 1` | `-` | `120` | 120 (CORRECT) | 120 |
| STARTING_FEN (5-field) | `w - 0 1` | `0` | `1` | 1 (WRONG) | 0 |

Detection rule: if parts[3][0] is a digit OR parts[3] == '-', it's WXF 5-field (no en passant). Otherwise, it's standard 6-field (en passant present at parts[3]).

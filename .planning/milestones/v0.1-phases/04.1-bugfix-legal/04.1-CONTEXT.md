# Phase 4.1: Bugfix — is_legal() piece-type validation + performance

**Gathered:** 2026-03-21

<domain>
## Phase Boundary

Fix two bugs discovered post-Phase-04 verification:

1. **`is_legal()` missing piece-type geometry check** — currently only validates in-check and flying-general, does NOT reject geometrically illegal moves (e.g., diagonal chariot moves, own-piece captures).
2. **`legal_moves()` too slow** — averages 29ms vs 10ms target, making RL training loops impractical.

</domain>

<decisions>
## Bugs to Fix

### Bug 1: is_legal() missing piece-type validation (MAJOR)

**Root cause (from code trace, legal.py:228-252):**
`is_legal_move()` only checks:
- Own piece on source square
- Not leaving own general in check
- No flying-general violation

It does NOT check that the move geometry is valid for the piece type (e.g., chariot must move orthogonally, horse must move in L-shape).

`generate_legal_moves()` does it correctly — it calls piece-specific generators (`gen_chariot`, `gen_horse`, etc.) which produce only geometrically valid moves, then filters with `is_legal_move()`. So `legal_moves()` correctly excludes diagonal rook moves.

The fix: Add piece-type geometry validation to `is_legal_move()` OR make `is_legal()` delegate through `generate_legal_moves()`.

### Bug 2: legal_moves() at 29ms, target 10ms

**Root cause (from profiling, #7129):** `generate_legal_moves()` iterates all pieces, calls piece generators, then filters each with `is_legal_move()`. For each candidate move, it copies the full board state with `state.copy()` (a `numpy.copy()` operation). At the starting position there are ~44 legal moves but hundreds of pseudo-legal candidates — each requiring a full board copy + check simulation.

**Fix approach:** Pre-filter pseudo-legal moves by checking destination before doing the expensive board-copy + is_in_check simulation. Skip the copy if the destination is blocked by own piece.

### Bug 3: undo() test expectation mismatch (NOT a bug)

**Note:** `04-VERIFICATION.md` reports test #5 as "issue". The implementation correctly raises `IndexError("nothing to undo")` per `04-CONTEXT.md`. The UAT test expectation (no-op) contradicts the spec. No code change needed — test expectation needs to be corrected.

</decisions>

## Bug 1: is_legal() Missing Piece-Type Geometry Validation

### Evidence
From `legal.py:228-252`, `is_legal_move()` only checks:
1. Own piece on source square
2. Not leaving own general in check (`is_in_check(snap, moving_color)`)
3. No flying-general violation

It does NOT validate move geometry matches piece type. This means diagonal chariot moves, L-shape rook moves, etc. all return `True`.

### Key observation
`generate_legal_moves()` handles this correctly by calling piece-specific generators (`gen_chariot`, `gen_horse`, etc.) which only produce geometrically valid moves. The filtered list `[m for m in moves if is_legal_move(state, m)]` correctly excludes geometry violations.

### Fix approaches

**Approach A — Add geometry check to is_legal_move():**
Validate move geometry directly in `is_legal_move()` using piece-type-specific rules before the board-copy simulation. This is more efficient than Approach B.

**Approach B — Delegate is_legal() through generate_legal_moves():**
`is_legal()` could check if the move appears in `legal_moves()`. Simple but O(n) per call.

**Approach A is recommended.** The fix: extract piece-type from the `from_sq` piece, then validate that the destination `(to_row, to_col)` satisfies the piece's movement rules before doing the expensive board-copy simulation.

### Relevant files
- `src/xiangqi/engine/moves.py` — piece generators already implement geometry rules
- `src/xiangqi/engine/legal.py` — is_legal_move() needs geometry check
- `src/xiangqi/engine/engine.py` — is_legal() delegates here

## Bug 2: legal_moves() Performance (29ms vs 10ms target)

### Evidence
From memory #7129: `legal_moves()` at starting position averages 29ms over 100 iterations. The bottleneck is the board-copy in `is_legal_move()` for every pseudo-legal candidate.

### Root cause analysis
`generate_legal_moves()` flow:
1. Iterate all pieces of current color
2. For each piece, call piece-specific generator → produces pseudo-legal destinations
3. For each candidate, call `is_legal_move()`:
   - Copy board state: `snap = state.copy()`
   - Apply move to snap: `apply_move(snap, move)`
   - Check in_check: scan entire board for attackers
   - Check flying_general_violation: scan column

At starting position: ~44 legal moves, but hundreds of pseudo-legal candidates. Most fail at step 1 (not own piece, own-piece destination) BEFORE the board copy — but the board copy is done inside `is_legal_move()`, after the source-square ownership check.

Wait — `is_legal_move()` does the source-square check BEFORE `snap = state.copy()` (line 243). So most candidates fail fast. The board copy is only done for moves that pass the source-square check.

But even with fast rejection, the loop still processes many candidates. The 29ms likely comes from:
- The numpy board copy operation (deep copy of 10x9 array)
- The `apply_move()` call (board mutation + hash update)
- The `is_in_check()` scan (full board scan in 4 directions)

### Fix approaches

**Approach A — Pre-filter destinations before is_legal_move():**
Before calling `is_legal_move()` for a candidate, check if `dest_square` contains own piece. This avoids the expensive board copy entirely for blocked destinations.

**Approach B — Caching:**
Cache `legal_moves()` result and invalidate on apply/undo. Simple but changes semantics.

**Approach C — Vectorized board copy:**
Replace `state.copy()` with a more efficient approach, or share immutable board representations.

**Approach A is recommended.** The `generate_legal_moves()` caller has the board and can check `board[dest] < 0` (for red's turn) before calling `is_legal_move()`. Most pseudo-legal destinations will be own-piece blocks or empty, and this avoids the board copy entirely.

### Relevant files
- `src/xiangqi/engine/legal.py` — generate_legal_moves() is the bottleneck
- `src/xiangqi/engine/state.py` — state.copy() implementation
